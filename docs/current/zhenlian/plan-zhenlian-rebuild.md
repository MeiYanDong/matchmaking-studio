## 二、与当前项目的关系

### 2.1 v1.0 已沉淀的基础能力

当前项目在 v1.0 阶段已经沉淀出这些核心能力：

- 红娘端桌面工作台
- 上传音频
- 音频转录
- 结构化提取
- 写回客户档案
- 匹配、补问、提醒、跟进

它当时优先解决的是：

- 建档效率
- 字段完整度
- 核心工作流稳定性
- 红娘端高频操作体验

这些能力仍然有效，但它们不再单独定义当前主线的产品边界。

### 2.2 当前主线在做什么

当前主线不是继续微调旧流程，而是把产品升级成：

**面向多红娘、多门店、多角色协同的婚恋业务操作系统。**

它不再只关注：

- 音频转录
- AI 建档
- 客户详情

而是扩展到：

- 数据生命周期管理
- 匹配解释与反馈闭环
- 多红娘协作与跨店撮合
- 分润与结算
- 销售工作流
- 认证体系
- 渠道追踪
- 活动系统
- 多门店中台
- SaaS 化潜力

### 2.3 接入策略

当前主线的默认策略不是一次性推翻既有实现，而是：

1. 继承已经跑通的上传、转录、提取、写库等既有基础能力
2. 把甄恋重构文档体系提升为当前主线 source of truth
3. 后续按模块决定：
   - 是逐步吸收到现有项目
   - 还是以更大范围的版本升级方式落地

也就是说：

**这条计划现在就是当前主线，不再是“独立规划、未来再接入”的副线。**

## 三、重构北极星

### 3.1 核心目标

把产品从：

**低效的数据录入与跟进工具**

升级为：

**围绕客户资产、匹配闭环、协作与分润运转的业务增长操作系统**

### 3.2 成功后的产品形态

成功后的甄恋 CRM，不只是能“记录客户”，而是要同时做到：

1. **看见客户生命周期**
   - 客户当前在什么阶段
   - 下一步应该做什么
   - 谁负责
   - 什么时候截止

2. **看见匹配为什么成立**
   - 不只给分
   - 还给依据、冲突、待确认点和推进建议

3. **支持多人协作**
   - 同门店协作
   - 跨门店协作
   - 资源申请
   - 脱敏共享

4. **支撑业务经营**
   - 渠道 ROI
   - 红娘绩效
   - 分润结算
   - 活动转化
   - 多门店视角

5. **形成长期护城河**
   - 数据资产
   - 匹配反馈回路
   - 认证体系
   - 话术库
   - 平台型交易与协作机制

## 四、当前聚合版方案要解决的三个结构性问题

### 4.1 数据没有生命周期

当前很多婚恋 CRM 的问题是：

- 客户档案只是静态表单
- 没有阶段、负责人、截止时间、下一步动作
- 客户推进主要靠红娘记忆

这会导致：

- 客户一多就乱
- 跟进动作不连续
- 系统无法提醒真正该做的事
- 管理层也看不清流程堵点

### 4.2 匹配是单向黑箱

当前很多系统只会输出：

- 匹配分

却回答不了：

- 为什么高分
- 为什么不能推
- 双方是否互相都合适
- 哪些地方需要补问
- 见面后反馈有没有回流到模型

这会导致：

- 红娘无法解释推荐依据
- 客户不容易被说服
- 匹配质量无法持续提升

### 4.3 系统不是协作平台

当前系统如果只服务单个红娘，规模一上来就会暴露：

- 客户归属不清
- 跨红娘资源无法流通
- 联系方式泄露风险
- 分润不清
- 总部与门店无法形成中台视角

因此重构后的系统必须从单人工具升级为：

**有权限边界、有协作流转、有交易和结算机制的平台。**

## 五、总的产品重构原则

### 5.1 客户档案必须从“表单”变成“数据资产”

档案不只是填写字段，而是要具备：

- 优先级
- 可信度
- 认证状态
- 生命周期状态
- 责任归属
- 可回溯来源

### 5.2 所有关键流程必须可解释

包括：

- 为什么推这个人
- 为什么还不能推
- 为什么要补问
- 为什么这条客户今天应该跟进
- 为什么这次成交要这样分润

### 5.3 AI 负责降本增效，不负责夺权

AI 在这条重构轨里的角色依然是：

- 自动整理
- 自动提示
- 自动推荐
- 自动归纳
- 自动提醒

但关键判断仍由人做：

- 是否推进
- 是否协作
- 是否成交
- 是否线下补问
- 是否通过认证

### 5.4 平台要天然支持协作、经营和扩张

重构后系统的目标不只是“好用”，而是：

- 单门店可用
- 多红娘可协作
- 多门店可扩张
- 未来能形成 SaaS 化能力

## 六、目标产品结构（六大核心模块）

## 6.1 模块一：客户档案系统重构

### 目标

把客户档案从“静态输入页面”改成“结构化数据资产中心”。

### 核心内容

1. 字段优先级重排
2. 动态联动表单
3. 认证体系
4. 数据导入多通道
5. 照片与材料上传体验优化
6. 关键字段标准化

### 必须落地的能力

#### A. 字段优先级分层

要把客户字段明确分为：

- P0 核心识别
- P1 匹配关键
- P2 条件筛选
- P3 匹配深度
- P4 认证资料
- P5 AI 分析

这不是简单分组，而是决定：

- 哪些字段第一屏必见
- 哪些字段影响匹配
- 哪些字段适合在详情深层展示
- 哪些字段由 AI 辅助生成

#### B. 字段逻辑修复

聚合版方案明确指出当前必须修复的问题包括：

- 学历字段要拆成“学历级别 + 院校”
- 收入字段要去重和重构
- 吸烟 / 饮酒字段去重
- 颜值评分标准化
- 房车资产拆分
- 孩子字段联动数量与抚养情况
- `认真程度` 改为更可执行的 `迫切程度`

#### C. 动态联动表单

系统要根据客户意图动态显示不同字段：

- 恋爱意图
- 结婚意图
- 生育意图
- 有孩子的联动字段

目的不是让表单更复杂，而是：

**只在真正需要的时候呈现更深入的信息。**

#### D. 认证体系

必须新增：

- 身份认证
- 学历认证
- 经济认证
- 婚况认证
- 健康认证

并让认证状态影响：

- 档案可信度
- 匹配优先级
- 推介说服力

#### E. 多通道导入

客户信息不能只靠手填，要支持：

- PDF 拖入识别
- 录音转写 + 字段提取
- 微信截图识别

这一步会直接影响建档效率。

### 对当前项目意味着什么

这一模块将要求未来重新审视：

- `profiles` 字段结构
- 认证相关表
- 生活习惯、意图、资产等字段拆分方式
- 导入入口与上传流
- 照片 / 认证材料管理

---

## 6.2 模块二：匹配引擎重构

### 目标

把匹配从“黑箱打分”升级为“可解释、可反馈、可迭代的双向匹配系统”。

### 核心内容

1. 三层混合匹配架构
2. 匹配结果可视化
3. 社交冲突检测
4. 红娘介入节点
5. 客户状态流转

### 必须落地的能力

#### A. 三层匹配架构

匹配引擎需同时具备：

- 第一层：规则硬筛
- 第二层：向量相似度软匹配
- 第三层：历史反馈学习

也就是说：

系统不能只算一个总分，而必须知道：

- 先筛掉不可能的
- 再排序可能的
- 再利用历史反馈不断修正

#### B. 分维度解释

匹配结果至少要拆解为：

- 外貌契合度
- 条件匹配度
- 生活方式相容
- 目标一致性
- 隐性化学值

每一维都要给解释，不然红娘无法对外表达。

#### C. 社交冲突检测

匹配前要检测：

- 客户社交关系冲突
- 介绍链路重叠
- 已见面组合冲突

这一步在高净值和熟人网络场景里很关键。

#### D. 反馈闭环

见面后反馈不能只停留在跟进记录中，而要反向进入匹配系统。

最终目标是：

**匹配结果会因为真实见面反馈越来越准。**

### 对当前项目意味着什么

这一模块会影响：

- `matches` 结构
- score breakdown 设计
- 见面反馈结构
- 匹配解释 UI
- 匹配重跑机制
- 后续模型训练数据积累

## 6.3 模块三：多红娘协作平台

### 目标

把系统从“单红娘工具”升级为“有权限、有协作、有资源流通”的平台。

### 核心内容

1. 权限分层
2. 跨红娘撮合市场
3. 联系方式保护
4. 协作申请与审批
5. 协作后分润

### 必须落地的能力

#### A. 角色与可见范围

至少要区分：

- 所属红娘
- 同门店同事
- 外地门店红娘
- 总部管理员

不同角色能看什么、能做什么，都要有明确边界。

#### B. 脱敏共享

协作前只能看脱敏资料，不能看联系方式。

#### C. 平台内交易与协作

跨红娘撮合不是聊天协作，而是一个平台流程：

- 资源方
- 客户方
- 申请
- 审批
- 价格
- 分润
- 平台抽成

### 对当前项目意味着什么

这一模块未来会新增：

- store / branch 概念
- collaboration request
- data visibility rule
- contact masking
- internal marketplace / deal records

## 6.4 模块四：销售工具系统

### 目标

让系统支持红娘从获客到签单的完整销售工作流，而不仅仅是匹配和建档。

### 核心内容

1. AI 话术库
2. 电话系统集成
3. 一键资料卡生成
4. 分润结算
5. 销售计划与行为轨迹

### 必须落地的能力

#### A. AI 话术库

基于历史录音和成交案例，生成可复用话术库：

- 初次邀约
- 价格异议
- 见面劝退
- 续费挽留

#### B. 电话工作台

包括：

- 一键外呼
- 通话录音
- 自动摘要
- 后期质量评分与监听

#### C. 资料卡生成

支持：

- 完整版
- 对客介绍版
- 营销版

#### D. 分润结算

要能自动处理：

- 销售红娘
- 交付红娘
- 跨店资源方
- 客户方
- 平台抽成
- 外部渠道引流

### 对当前项目意味着什么

这一模块未来会影响：

- 话术库数据模型
- call log / telephony 接入
- profile card 生成能力
- settlement / commission ledger
- 渠道与来源字段

## 6.5 模块五：数据与运营系统

### 目标

至少应支持：

- 来源
- 成交额
- CAC
- 留存与见面率

### 对当前项目意味着什么

未来会新增：

- 渠道表
- UTM / source code
- 经营看板

- 自动画像统计任务
