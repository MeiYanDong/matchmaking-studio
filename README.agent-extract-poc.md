# Matchmaking Studio - Agent 提取实验支线说明书

> 分支：`codex/agent-extract-poc`  
> 日期：2026-04-03  
> 适用对象：非技术同事、产品负责人、红娘业务负责人、合作方  
> 这份文档的目标：用尽量容易理解的话，说明我们为什么要重建这条核心处理链路、当前到底卡在哪里、以及这条新方案打算解决什么问题。

---

## 一、这条支线在做什么

这条支线不是在“做一个新产品”，而是在为当前产品最核心的工作流做一次重建实验。

当前产品最重要的一条主线是：

```text
上传一段录音
-> 把录音变成文字
-> 从文字里提取客户信息
-> 更新客户资料库
```

表面上看，这只是一个“上传录音”的功能。  
但实际上，这条链路决定了后面几乎所有能力能不能成立：

- 客户档案是否更新准确
- 匹配是否准确
- 红娘是否能少做手工整理
- AI 是否真的帮上忙

所以这条支线的本质是：

**不是改界面，而是重建这条核心“录音 -> 数据”的主路径。**

---

## 二、为什么要重建这条链路

因为现在这条链路虽然已经能跑，但还不够稳定，也不够适合真实业务使用。

我们已经验证过，当前方案的问题不在于“模型不够聪明”，而在于：

**我们让模型一次做了太多事。**

现在的做法大致是：

1. 用户上传音频
2. 系统把音频转成文字
3. 把整段文字、客户当前资料、历史对话、字段说明、系统规则一起发给 Claude
4. 要求 Claude 一次性返回一大段结构化 JSON
5. 系统再解析这段 JSON，写入数据库

这看起来很直接，但真正运行时会暴露很多脆弱点。

---

## 三、当前方案的痛点是什么

### 1. 一次性返回一个“大 JSON”，非常脆

目前 AI 不只是要“看懂文本”，还要一次性返回一份格式非常严格的结构化结果。

这就会带来很多现实问题：

- 返回内容太长
- JSON 容易损坏
- 某个字段格式不对，就可能整包失败
- 前面都做对了，最后因为一个格式问题还是不能入库

换句话说：

**现在的问题，很多时候不是 AI 没理解，而是它理解了，但返回格式太脆。**

### 2. AI 被要求同时做太多任务

当前 AI 不只是提字段，还要同时输出：

- 哪些字段可以写入
- 哪些字段需要人工确认
- 哪些关键字段还没问到
- 下一步该怎么补问
- 总结描述
- 处理备注

这就像让一个人一次性同时写：

- 会议纪要
- 行动清单
- 风险分析
- 待办事项
- 最终总结

任务本身不是不能做，但全压在一次性返回里，失败概率会明显上升。

### 3. 提取和写库绑得太死

现在这条链路的关键问题是：

- 只有当 AI 返回的 JSON 完全合格
- 系统才能继续写数据库

也就是说，写库是否成功，过度依赖“AI 这次返回得够不够漂亮”。

而业务真正需要的是：

**只要 AI 已经读懂了内容，就尽量把可用的字段先写进去。**

### 4. 调试困难，恢复困难

一旦当前流程出问题，排查会很痛：

- 到底是转录错了？
- 还是字段判断错了？
- 还是返回格式坏了？
- 还是数据库没写进去？

更糟的是，当前很多错误不是“判断错”，而是“格式坏”，这会让问题显得很随机，也很难让人信任。

### 5. Supabase 远程链路对本地实验不够友好

在正式产品里，Supabase 是合理的。  
但在实验支线里，Supabase 有两个问题：

- 远程速度慢，影响我们快速验证
- Agent 不方便直接用它做高频实验性编辑

所以如果继续沿用 Supabase 做这条 POC，会让我们把大量时间浪费在：

- 网络等待
- 远程存储
- 环境同步
- 权限绕路

而不是浪费在真正要验证的东西上。

---

## 四、我们真正想解决的，不只是一个报错

这次实验不是为了修一个具体的 bug，也不是为了“把 JSON 再修补一下”。

我们想解决的是更底层的问题：

### 问题 1：如何让 AI 结构化提取更稳定

目标不是让 AI 每次说得更漂亮，而是让它：

- 更稳定地把录音内容变成数据
- 更少因为格式问题失败

### 问题 2：如何让“写数据库”成为主目标，而不是副产物

现在真实业务最重要的不是“输出一段完美 JSON”，而是：

- 客户年龄写进去了没有
- 城市写进去了没有
- 婚恋意图写进去了没有
- 下一步补问有没有形成

所以系统应该围绕“稳定写库”来设计，而不是围绕“模型一次性返回合同”来设计。

### 问题 3：如何让系统更接近未来的云端 AI 助理

未来我们真正想要的不是一个“只能返回 JSON 的模型接口”，而是一个更像助理的系统：

- 读懂 transcript
- 读取已有客户档案
- 理解历史谈话
- 决定更新哪些字段
- 直接写入数据库
- 生成补问和摘要

这其实更像一个 Agent，而不是一个普通 API 调用。

---

## 五、这条新方案的核心思路是什么

核心思路可以概括成一句话：

**把“AI 一次性返回大 JSON”改成“Agent 分步骤处理并直接更新数据库”。**

旧方案更像这样：

```text
音频
-> 文字
-> 把所有上下文一次性发给 Claude
-> Claude 返回一大段 JSON
-> 系统解析 JSON
-> 写数据库
```

新方案更像这样：

```text
音频
-> 文字
-> Agent 读取文字稿
-> Agent 读取客户当前档案
-> Agent 结合历史对话和字段规则做判断
-> Agent 直接更新数据库
-> Agent 再生成摘要和补问
```

这两条路看起来都叫“AI 提取”，但结构完全不同。

---

## 六、为什么我们认为新方案更合理

### 1. 它更像真实的人类助理工作方式

人类红娘不会一次性先写一个超级完整的 JSON 再去更新系统。  
人类的真实做法是：

1. 先听录音
2. 先记下明确事实
3. 再结合已有档案修正
4. 最后补一句总结
5. 如果有不确定的地方，再列待确认问题

Agent 化方案更接近这个自然流程。

### 2. 它不要求“一次成功，全部正确”

当前方案的隐含要求是：

- 你要一次性理解对
- 一次性返回对
- 一次性格式也对

这很不稳。

Agent 化之后，系统可以拆成多步：

- 先更新明确字段
- 再更新推断字段
- 再写摘要
- 再生成补问

只要前面几步成功，就已经有实际业务价值。

### 3. 更容易调试和复盘

如果新方案出错，我们更容易知道是卡在哪一步：

- transcript 有问题
- 档案读取有问题
- 某个字段判断有问题
- 数据库写入有问题

这比“AI 返回格式解析失败”更可诊断，也更可信。

### 4. 更适合后续积累经验

以后这套系统一定会不断积累：

- 常见问法
- 常见术语
- 城市/学历/收入归一化规则
- 红娘补问经验
- 敏感字段的判断经验

这些东西非常适合沉淀到 Agent 流程里。  
如果还停留在“一次性 JSON 合同”模式，就会越来越重、越来越难维护。

---

## 七、这条支线为什么先不用 Supabase

不是因为 Supabase 不好，而是因为：

**它不适合当前这次 POC 的实验目标。**

这次实验的第一目标不是“先做线上兼容”，而是：

**先验证 Agent 能不能更稳地完成结构化提取和写库。**

所以我们决定：

- 这条支线先用本地数据库
- 不依赖远程 Supabase
- 不优先解决云端权限问题
- 先把核心逻辑在本地跑通

这样做的好处是：

- 速度更快
- 调试更直接
- Agent 更容易直接操作数据库
- 一旦方案成立，再迁回云端也更容易

一句话：

**先在本地验证模型和流程，后面再考虑上线架构。**

---

## 八、这条支线到底想验证什么

这条实验支线不追求一步到位，它只验证最关键的东西：

### 我们想验证的问题

1. Agent 能不能读懂整段 transcript  
2. Agent 能不能结合已有客户资料做增量更新  
3. Agent 能不能直接把字段写进本地数据库  
4. 这条路是不是比“大 JSON 合同”更稳  

### 我们暂时不追求的东西

- 不先追求生产级安全边界
- 不先追求云端部署完美
- 不先追求完整 UI
- 不先追求所有边缘场景

先把“核心主线能跑稳”放在第一位。

---

## 九、这条新方案打算怎么做

### 第一步：保留现有转录能力

这条支线不会推翻现在的转录逻辑。  
我们仍然会继续用：

- `Groq Whisper`

来完成音频转文字。

也就是说，真正变化的不是“录音怎么变文字”，而是：

**文字之后，怎么结构化提取并写库。**

### 第二步：把 transcript 当作输入，让 Agent 处理

在新方案里，Agent 会读取：

- transcript
- 当前客户档案
- 历史谈话摘要
- 字段规则

然后一步步处理，而不是返回一个一大坨 JSON。

### 第三步：让 Agent 直接编辑本地数据库

这次实验支线里，我们不再让 AI 只做“返回结果”，而是让它直接成为执行者：

- 判断字段
- 直接改数据库
- 生成摘要
- 生成补问

这能最大程度验证：

**Agent 化执行是不是比当前 API 调用式 JSON 返回更有效。**

---

## 十、为什么 POC 阶段不过度限制 Agent 能力

这是一个很重要的决定。

在正式生产环境里，当然应该：

- 做权限约束
- 做工具边界
- 做最小权限
- 做更强的审计

但这次是实验阶段。  
实验阶段最重要的是：

- 验证方案是否成立
- 验证成功率
- 验证调试效率

如果一开始就把 Agent 限得太死，可能会把真正想验证的能力一起阉割掉。

所以这次支线的策略是：

**先让 Agent 真能做事，先验证结构是否更优；等验证成功，再回头收权限。**

当然，这不意味着完全无序。  
实验阶段仍然应该保留：

- 日志
- 回放能力
- dry-run / real-run 区分

这样即使 Agent 能力更强，整个实验仍然是可控的。

---

## 十一、这个方案如果成功，会带来什么变化

如果这条支线验证成功，后续我们就有机会把当前最脆弱的一段重构掉。

可能带来的变化包括：

### 1. “AI 返回格式解析失败”这类问题会明显减少

因为系统不再把稳定性过度押在一个大 JSON 上。

### 2. 字段入库的成功率会更高

因为系统可以分步骤写库，而不是“一次性全成或全败”。

### 3. 红娘会感觉系统更像一个助理，而不是一个脆弱接口

也就是：

- 上传录音
- 系统就把资料更新了

而不是：

- 上传录音
- 等 AI 返回一大坨结构化 JSON
- 再看哪里炸了

### 4. 更容易迁移到云端 Agent

如果本地 POC 成功，后面迁到云服务器 / OpenClaw 时，主要是部署问题，不再是逻辑是否成立的问题。

---

## 十二、这条支线不是什么

为了避免误解，这条支线不是：

- 不是在推翻整个产品
- 不是在重做 UI
- 不是在做另一个平台
- 不是在放弃现有正式版本

它只是：

**为核心工作流寻找一个更稳、更像真实 AI 助理的执行结构。**

---

## 十三、我们如何判断这条支线值不值得继续

不是看“写出来是否复杂”，而是看这几个结果：

### 成功信号

- transcript 能更稳定地变成数据库字段
- 失败更少
- 出错更容易定位
- 本地调试更顺
- 对真实录音的处理质量更稳定

### 失败信号

- 虽然结构更高级，但成功率并没有提高
- Agent 直接写库反而更混乱
- 本地调试并没有比当前方案更简单
- 迁移到云端的成本明显超过收益

如果是前一种，说明这条支线值得继续。  
如果是后一种，我们就及时止损。

---

## 十四、当前结论

这条支线之所以成立，不是因为我们单纯想“尝试新技术”，而是因为当前正式方案已经暴露出一个很清楚的结构性问题：

**我们把 AI 提取做成了“高耦合、高脆弱、高解释负担”的一次性返回。**

而这条实验支线想做的，是把它改成：

**一个更接近真实助手工作方式的 Agent 执行链路。**

这就是为什么我们要重建这条核心路径。

---

## 十五、下一步会做什么

当前支线接下来的动作会是：

1. 在本地建立实验环境
2. 起本地数据库，不用 Supabase
3. 保留 Groq 转录
4. 让 Agent 直接根据 transcript 更新数据库
5. 用几条真实 transcript 回放测试
6. 和当前大 JSON 方案做对比

如果这一轮通过，再进入下一阶段：

- 迁到云端
- 接 OpenClaw
- 再考虑权限收敛和生产化

---

## 十六、一句话版本

**这条支线不是为了炫技，而是为了验证：用 Agent 直接处理 transcript 并写库，能不能比当前“Claude 返回大 JSON 再解析”的方案更稳、更适合真实业务。**
